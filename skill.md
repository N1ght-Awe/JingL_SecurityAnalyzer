# JingL 漏洞扫描Agent

## 版本标识

当前Skill版本：`1.1.0`；当前规则版本：`1.1.0`。每次修改主流程、共享引用、子Skill判定规则或报告字段时必须同步更新版本：补丁修订递增patch，新增兼容规则递增minor，改变结论语义或报告契约递增major。每份扫描报告必须记录这两个版本。

## 启动方式、范围与技术栈

本Skill由用户人工放入模型可读取的目录后触发。用户说“`jingl 扫描 <代码仓或根目录>`”即进入本流程；仅扫描用户提供的目标根目录及其子目录，不自行扩展到其他路径。

**当前正式覆盖范围：Java/Spring Boot/Jalor。** Java源码、XML Mapper、Spring配置、构建依赖和部署清单为默认分析对象。Python、Node、前端等规则仅保留为未来扩展参考，除非用户明确要求，否则不作为正式覆盖范围或“未发现即安全”的依据。

## 运行目标

将大量代码命中压缩为少量证据充分的漏洞发现，每条发现具备清晰的证据链、利用前置条件、置信度和调用链。

**核心质量目标：执行完毕后报告的有问题，是真的有问题。** 宁可多花时间确认，不可产出"可能有问题"的模糊结论。结论只有三种：确认(真有问题)、误报(确认没问题)、缺失关键源码(证据不足，不判确认也不判误报)。

默认基于源码证据进行分析。运行时或Payload验证仅在明确授权的测试环境中进行。

## 执行模式

根据项目规模选择执行模式。两种模式的核心区别在于**谁负责判定**——并行Agent只做"发现和追踪"，主Agent做"判定和验证"。

### 串行模式（单模块或Java文件<500）

主Agent按阶段1→9顺序执行所有阶段。子Skill在阶段5由主Agent加载。

### 运行验证模式（所有规模适用）

- **source-only（默认）**：仅使用源码、配置和依赖证据；传输结论只能标记为“源码分析推断”，不得声称已拦截或已利用。
- **authorized-tool-assisted**：仅在用户明确授权的测试环境中，按 `security-tool-probing.md` 使用工具验证；报告必须记录授权范围、工具、时间和原始证据位置。

两种模式的静态扫描流程相同；差别仅在阶段7能否获得工具验证证据。

### 并行模式（按规模与复杂度触发）

**阶段职责分配**：

| 阶段 | 执行者 | 说明 |
|------|--------|------|
| 阶段1：项目画像 | 主Agent | 构建全局画像 |
| 阶段2：输入源扫描计划 | 主Agent | 基于全局画像生成 |
| 阶段3：轻量候选发现 | **并行Agent** | 按scan-index关键词Grep搜索 |
| 阶段4：候选排序 | **并行Agent** | 初步排序，标注防护名称但不判定强弱 |
| 阶段5：加载子Skill深度分析 | **主Agent** | 加载完整子Skill，逐条审查P0/P1候选 |
| 阶段6：调用链深度追踪 | **并行Agent** | 双向追踪，返回完整调用链 |
| 阶段6.5：跨代码仓调用链追踪 | **主Agent** | 需要全局视图 |
| 阶段7：传输链路探测 | **主Agent** | 执行敏感数据反向追踪 |
| 阶段8：证据闭合验证 | **主Agent** | 8维度逐一确认 |
| 阶段9：去重与报告 | **主Agent** | 统一去重和生成报告 |

**为什么并行Agent不做判定**：
- 并行Agent无法加载子Skill，缺少完整判定标准和误报模式
- 并行Agent无法看到其他代码仓的上下文，防护分类可能不准
- 判定需要读取防护实现代码（如fileNameCheck、OgnlUtil），并行Agent可能跳过

**并行决策**：扫描根目录含3个及以上子代码仓，或单仓生产Java文件超过500时必须并行。低于阈值但入口数量、模块数量、跨仓调用点或候选数量明显较多时，主Agent也可按模块并行；选择依据必须记录在扫描统计中。

**并行模式执行流程**：
1. 主Agent完成阶段1-2
2. 主Agent分派并行Agent执行阶段3+4+6
3. 并行Agent返回结构化候选清单（见"并行Agent输出格式"）
4. 主Agent执行阶段5：按漏洞类型逐个加载子Skill，对P0/P1候选深度审查
5. 主Agent执行阶段6.5：跨代码仓关联分析
6. 主Agent执行阶段7：传输链路探测（含敏感数据反向追踪）
7. 主Agent执行阶段8：证据闭合验证
8. 主Agent执行阶段9：去重与报告

## 扫描工作流（9个主阶段 + 阶段6.5，共10个执行关卡，不可跳过）

### 阶段1：项目画像

在漏洞扫描之前，先检查构建和配置文件，识别框架、输入源、安全控件、解析器、数据访问层、依赖风险和项目边界。

按 `repo-boundary-manifest.md` 为每个代码仓记录服务名、入/出站接口或Topic、共享存储、共享凭据引用和证据位置；未知目标仓必须明确记录为`unknown`。

检查以下文件（如存在）：
- `pom.xml`、`build.gradle`、`settings.gradle`
- `package.json`、lockfiles
- `application.yml`、`application.properties`、`bootstrap.yml`
- Docker、compose、Kubernetes清单
- Spring Security、Shiro、Filter、拦截器、网关配置
- Nacos、Apollo、配置中心引用

记录每条画像事实：
```
项目:
来源文件:
作用范围:
置信度:
全局适用性:
局限性:
```

**全局画像事实仅作上下文参考，不能作为具体finding的误报证明，除非映射到确切的代码路径。**

### 阶段2：生成输入源扫描计划

根据阶段1的项目画像证据，确定启用哪些输入源扫描。

| 项目证据 | 启用的输入源 |
|---------|------------|
| `spring-boot-starter-web`、MVC注解 | `@RestController`、`@Controller`、映射注解、`HttpServletRequest`、请求头、Cookie、请求体 |
| `spring-boot-starter-webflux` | 响应式Controller、`RouterFunction`、`ServerRequest` |
| `MultipartFile`、上传配置 | 文件上传内容、原始文件名、导入管道 |
| `spring-kafka` | `@KafkaListener`、`ConsumerRecord`、消息载荷 |
| RabbitMQ依赖 | `@RabbitListener`、`Message`载荷 |
| RocketMQ依赖 | `@RocketMQMessageListener`、`onMessage` |
| Dubbo依赖/配置 | `@DubboService`、服务XML、RPC方法参数 |
| gRPC依赖 | 生成的服务实现、请求消息 |
| Quartz、XXL-JOB、`@Scheduled` | 定时任务参数、Handler execute、调度器控制的输入 |
| WebSocket依赖 | `@MessageMapping`、`WebSocketHandler`、`TextMessage` |
| GraphQL依赖 | `@QueryMapping`、`@MutationMapping`、GraphQL解析器 |
| CSV/Excel/XML/JSON解析器 | 导入文件内容及解析字段 |
| ORM/数据访问层 | 数据库二次注入（存储数据后续到达Sink） |

如果画像不完整，保守扫描常见输入源，并在扫描统计中标注"项目画像不完整"。

对每个输入源记录：
```
源类型:
位置:
参数或字段:
数据类型:
是否需要认证:
初始可控性:
初始利用前置条件:
```

### 阶段3：轻量候选发现

使用 `shared-references/scan-index.md` 中的四层规则，通过Grep、结构化搜索、配置/依赖核验和数据流追踪发现候选。此阶段仅创建候选，不产出最终漏洞发现。

按 `scan-index.md` 中17种漏洞类型的关键词逐类型搜索，每条命中记录：
```
漏洞类型:
位置:
匹配规则:
上下文:
源证据:
Sink证据:
```

### 阶段4：候选排序

每条命中必须排序为P0/P1/P2/P3，不得静默忽略大量命中。

| 等级 | 含义 | 必须处理 |
|------|------|---------|
| P0 | 源和Sink直接或短路径连接，无可见强防护 | 主Agent加载对应子Skill并完整审查 |
| P1 | 危险Sink存在，源需追踪或防护不明确 | 主Agent加载对应子Skill并继续追踪 |
| P2 | 因枚举/配置/服务间调用/可能的白名单而优先级较低，但未证明安全 | 记录原因，采样或在风险值得时继续 |
| P3 | 仅记录：固定字面量、测试/演示/生成代码、证明不可控、证明有强防护 | 汇总排除原因 |

**并行模式下，排序依据仅基于可见证据，不判定防护强弱**：
- 看到`OgnlUtil.getOrderType()` → 标注防护名称"OgnlUtil.getOrderType()"，排序为P2（待主Agent确认是否强防护）
- 看到`fileNameCheck` → 标注防护名称"fileNameCheck"，排序为P1（待主Agent读取实现后判定）
- 看到硬编码常量 → 排序为P3
- 无任何可见防护 → 排序为P0

**P0/P1准入门槛（减少低质量候选进入深度审查）**：

候选必须同时满足以下条件才能标记为P0或P1，否则默认降为P2：

| 准入条件 | P0要求 | P1要求 |
|---------|--------|--------|
| 用户可控证据 | 有明确的用户输入入口证据（Controller参数、请求头、MQ消息等） | 输入可能可控，需追踪确认 |
| Sink确认 | 危险操作已确认存在（如`${}`在XML中、`Runtime.exec`等） | Sink存在但需确认参数来源 |
| 防护状态 | 无任何已知强防护名称 | 有防护名称但未读取实现 |

**不满足准入门槛的降级示例**：
- 候选仅有`${}`但无法确认用户可控 → 降为P2
- 候选有`Runtime.exec`但参数为硬编码常量 → 降为P3
- 候选有`new File(path)`但path来自@Value配置 → 降为P2

**间接输入例外通道**：

当Sink的数据源是以下"间接输入"时，即使无法在当前代码仓追溯到最终用户入口，也可标为P1（因为数据来自外部存储/消息，本身就是不可信源）：

| 间接输入类型 | 说明 | 准入处理 |
|-------------|------|---------|
| 数据库读取后的二次注入 | 用户输入先写入数据库，后续从数据库读取后到达Sink（如存储型XSS、SQL二次注入） | 标P1，标注"数据库二次注入，原始入口未追溯" |
| MQ消息消费 | 消息来自另一个服务，当前代码仓只能看到消费端 | 标P1，标注"MQ消息源不可见，生产端需跨仓追踪" |
| 文件/缓存读取 | 数据来自文件系统或缓存，可能由其他服务写入 | 标P1，标注"文件/缓存数据源，写入端需追踪" |
| 跨服务RPC调用参数 | Dubbo/gRPC等RPC参数来自调用方服务 | 标P1，标注"RPC调用方不可见，需跨仓追踪" |

**例外通道的核心原则**：数据库、消息队列、文件系统、RPC都是信任边界外的数据源。即使当前代码仓看不到数据的生产者，也不能假设数据是可信的——必须按不可信源处理，标P1并说明缺失环节。

**结论继承规则（减少重复审查）**：

当多条候选满足以下全部条件时，后续候选直接继承首条已审查候选的结论，无需重复审查：
1. 调用链相同（从同一入口到同一Sink的路径一致）
2. 问题本质相同（同一漏洞类型、同一危险模式）
3. 防护状态相同（同一种防护覆盖所有候选，或均无防护）

**严重等级取最高**：如果继承的候选与原候选严重等级不同，继承时取最高严重等级。例如原候选P2、继承候选因利用前置条件更宽松（如无需认证）应为P1，则继承后统一标为P1。严重等级评估应考虑所有继承候选中利用前置条件最宽松的情况。

继承格式：
```
候选ID: [继承候选ID]
继承自: [原候选ID]
继承原因: 调用链相同+问题相同+防护相同
继承结论: [确认/误报/缺失关键源码]
严重等级: [取所有继承候选中最高的等级]
严重等级说明: [为什么取该等级，如"候选B无需认证，前置条件最宽松"]
```

对每种漏洞类型统计：
```
漏洞类型:
总命中数:
P0数:
P1数:
P2数:
P3数:
处理摘要:
```

同时更新 `coverage-metrics.md` 规定的量化统计；“候选为0”只能说明当前覆盖范围内未命中，不能说明该类型安全。

### 阶段5：加载子Skill并深度分析

**仅对存在P0/P1候选的漏洞类型**，加载 `skill/` 下对应的子Skill，获取完整的危险模式清单、判定标准、误报模式，然后深度分析。

对每条P0/P1候选：
1. Read上下文（前后各30行）
2. 结合子Skill判定标准分析，并按 `subskill-contract.md` 补齐该候选的最小判定字段
3. **读取候选中提到的防护实现代码**（如fileNameCheck、OgnlUtil、checkInTrustEnum等），按sanitizers.md分类为强防护/弱防护/未知防护
4. 执行调用链追踪（见阶段6）
5. 按子Skill判定标准给出最终结论：确认/误报/缺失关键源码

**并行模式下，此阶段由主Agent在收到并行Agent结果后执行**。主Agent按漏洞类型逐个加载子Skill，对并行Agent返回的P0/P1候选逐一深度审查。

### 阶段6：调用链深度追踪

**最关键的阶段。** 对每条P0/P1候选执行双向追踪：

**正向追踪（从入口到危险点）**：
1. 从阶段2发现的输入源开始
2. Grep搜索Controller方法名，找到调用的Service
3. Grep搜索Service方法名，找到调用的DAO/Repository
4. 逐层追踪直到危险操作点

**反向追踪（从危险点到入口）**：
1. 从Grep命中的危险函数开始
2. Read危险函数所在方法，记录方法签名
3. Grep搜索该方法被谁调用
4. 逐层向上追踪直到输入源或确认无用户可控路径

**追踪规则**：
- 每条finding至少追踪2层调用链
- 追踪到输入源且参数用户可控 → 标记"用户可控"
- 追踪到硬编码常量/枚举/配置 → 标记"不可控"
- 追踪2层后仍无法确定来源 → 标记"缺失关键源码"，**不得直接判误报**

**并行Agent最小追踪深度约束（强制）**：

并行Agent在阶段3+4+6中必须遵守以下最小追踪深度，**不得以"Grep结果太多"或"时间有限"为由跳过追踪**：

| 候选等级 | 最小正向追踪深度 | 最小反向追踪深度 | 说明 |
|---------|----------------|----------------|------|
| P0 | 到达输入源或3层（取较浅者） | 2层 | P0必须有用户可控证据或明确标注"缺失" |
| P1 | 2层 | 1层 | P1必须确认Sink参数来源 |
| P2 | 1层 | 0层 | 仅记录上下文，不做深度追踪 |
| P3 | 0层 | 0层 | 仅记录排除原因 |

**追踪豁免条件**：只有以下情况可以不满足最小追踪深度：
- 已通过源码确认问题不存在（如参数为硬编码常量、防护已确认有效）
- 代码仓不包含追踪目标文件（标注"缺失关键源码"并说明缺失环节）

**并行分支数量控制**：为减少主Agent审查压力，并行Agent数量不超过代码仓子项目数量，每个Agent负责1-2个子项目。**不得为每种漏洞类型单独分派Agent**——这会导致候选数量爆炸式增长。

**追踪深度要求**：

| 场景 | 最少追踪层数 |
|------|------------|
| 危险点在Controller中 | 1层 |
| 危险点在Service中 | 2层 |
| 危险点在DAO/Repository中 | 3层 |
| 危险点在工具类/Delegate中 | 3层+ |

**追踪结果记录格式**：
```
输入源: Controller.method(@RequestParam String param)
  → Service.method(param)
    → [危险操作] param未校验直接使用
```

如果追踪不完整，必须标注"缺失关键源码"及具体缺失环节。

### 阶段6.5：跨代码仓调用链追踪

**微服务/多代码仓架构的必要步骤。** 现代业务系统通常由多个代码仓组成，上传接口在代码仓A，执行逻辑在代码仓B，这种跨代码仓的调用链是漏洞分析常见类型。仅追踪单个代码仓内部调用链会遗漏大量跨代码仓漏洞。

**触发条件**（满足任一即执行）：
- 扫描目标包含2个以上子项目/代码仓
- 阶段1项目画像发现服务间调用配置（如`application.properties`中引用其他服务端点）
- 阶段6调用链追踪到达代码仓边界（方法调用指向外部服务/接口）

**步骤1：跨代码仓调用关系发现**

从配置文件和代码中提取跨代码仓调用证据：

```
搜索目标:
1. application.properties/yml中的外部服务URL配置（如 ${hdp.server}/copilotexecutor/*）
2. Feign Client/RestTemplate/WebClient调用外部服务
3. 消息队列（MQS/Kafka/RabbitMQ）的Topic配置和消费者
4. WebSocket连接配置
5. MCP/Agent通信协议配置
6. S3/OBS等共享存储配置
7. 共享认证系统（SSO/IDAAS/OAuth）的调用链
```

对每条跨代码仓调用记录：
```
调用方代码仓: 
被调用方代码仓:
通信协议: HTTP/MQS/WebSocket/MCP/S3/SSO等
调用接口: 具体的URL路径或Topic
传输数据: 认证凭据、用户输入、文件内容等
安全机制: Token传递方式、SSL验证状态、签名校验等
```

**步骤2：跨代码仓漏洞链构建**

将不同代码仓的finding通过跨代码仓调用关系串联，构建漏洞链：

```
跨代码仓漏洞链构建规则:
1. 代码仓A的finding（如文件上传无校验）→ 代码仓B的finding（如路径穿越）= 组合漏洞链
2. 代码仓A的传输安全finding（如verify=False）→ 代码仓B的凭据使用 = 凭据横向扩展链
3. 代码仓A的认证finding（如无SSO校验）→ 代码仓B的敏感操作 = 越权链
4. 多个代码仓共享同一凭据/密钥 → 一个被攻破全部受影响 = 凭据横向扩展链
5. 代码仓A的输入 → 消息队列 → 代码仓B的消费处理 = 异步攻击链
6. 代码仓A的Skill/插件下载 → 代码仓B的执行 = 供应链攻击链
```

**步骤3：跨代码仓调用链深度追踪**

对每条跨代码仓漏洞链，执行深度追踪：

```
追踪维度:
1. 数据流追踪: 用户输入 → 代码仓A → 序列化/传输 → 代码仓B → 反序列化/处理 → 危险操作
2. 凭据流追踪: 认证Token/Cookie → 代码仓A → HTTP头/Cookie → 代码仓B → 权限校验
3. 传输安全追踪: 代码仓间通信的SSL/TLS验证状态、协议安全性
4. 横向扩展追踪: 一个代码仓的凭据泄露如何影响其他代码仓
```

**步骤4：跨代码仓漏洞链记录格式**

```
链路ID: XC-001
链路名称: 代码仓A→代码仓B HTTP API
调用方: 代码仓A(技术栈)
被调用方: 代码仓B(技术栈)
协议/通信方式: HTTP/MQS/WebSocket等
传输数据: 具体传输的敏感数据
安全风险点: 关联的finding编号列表
跨代码仓漏洞链: 完整攻击路径描述
攻击入口: 攻击者从哪个代码仓进入
攻击影响: 跨代码仓攻击的最终影响
严重等级: P0/P1/P2
修复建议: 跨代码仓的修复方案
```

**核心原则**：
- 跨代码仓调用链追踪不是可选步骤，是微服务架构下漏洞扫描的基本逻辑
- 一个代码仓的输入验证漏洞，可能通过跨代码仓调用链在另一个代码仓中被放大为RCE
- 传输安全（verify=False、http://）在跨代码仓场景下影响倍增——攻击者只需攻破一个链路节点即可横向扩展
- 共享凭据/密钥的多个代码仓必须评估凭据横向扩展风险

### 阶段7：安全工具辅助传输链路探测

**网络安全工程师视角的必要步骤。** 调用链追踪（阶段6）关注代码内部数据流，但漏洞的利用往往发生在传输链路上。本阶段使用安全工具（Burp Suite、mitmproxy、Wireshark、OWASP ZAP等）验证传输链路上的安全风险。

读取 `shared-references/security-tool-probing.md` 获取完整的工具矩阵、探测维度和检查项。

**核心原则**：
- 工具是验证手段，不是目的——补充源码分析的不足
- 不限于Burp Suite——根据场景选择最合适的工具
- 不限于敏感信息泄露——覆盖认证绕过、会话劫持、请求篡改、CORS滥用、SSL降级等
- 源码优先，工具验证——先基于源码识别风险点，再用工具验证可利用性

**步骤1：传输链路清单构建**

从阶段6的调用链追踪结果中，提取所有涉及网络通信的节点，构建传输链路清单：

```
链路ID: TL-001
源端: 应用内部（认证模块）
协议: HTTPS
目标: IDaaS SSO服务
传输数据: mtoken、authToken、Cookie
SSL验证: verify=False
代理支持: 是（系统代理）
风险标记: SSL验证关闭 + 凭据传输
```

**步骤2：传输链路风险扫描**

在阶段3的Grep结果基础上，增加传输链路维度的源码信号扫描（关键词见scan-index.md），识别：
- SSL/TLS证书验证关闭（`verify=False`、`ignore_https_errors`等）
- 明文协议使用（`http://`、`ws://`）
- 不安全的Cookie/Token处理（缺少Secure/HttpOnly/SameSite）
- CORS配置问题（`allow_origins=["*"]` + `allow_credentials=True`）
- 代理信任配置（`HTTP_PROXY`、系统代理）

**步骤3：敏感数据反向追踪（强制）**

**发现以下信号时必须执行反向追踪，不能只记录信号本身**：

触发条件：
1. `verify=False` / `hostname.verify.ignore=true` / `ssl_verify=False`
2. `http://` 或 `ws://` 外部通信端点（非localhost）
3. RestTemplate/HttpClient/Feign 调用外部服务

追踪步骤：
1. 识别HTTP客户端实例——找到信号所在代码，识别对应的HTTP客户端类和变量名
2. 追踪该客户端的所有请求——Grep搜索客户端变量名，找到所有使用该客户端发起请求的位置
3. 识别传输的敏感数据——对每个请求，检查请求头（Authorization/Cookie/X-Auth-Token）和请求体是否包含凭据/Token/密码
4. 评估拦截影响——如果该请求被拦截，攻击者获得什么？获得的凭据是否可用于横向攻击？

追踪结果格式：
```
传输安全信号:
  信号类型: SSL验证关闭 / HTTP明文 / hostname验证关闭
  代码位置: 文件:行号
  影响的HTTP客户端: [具体类名和变量名]

反向追踪结果:
  客户端实例: [类名.变量名]
  请求1: [方法名:行号] → [HTTP方法] [目标URL模式], 请求头: [Authorization/Cookie/无], 请求体含敏感字段: [是/否, 具体字段]
  请求2: ...

传输的敏感数据汇总: [Token/密码/API Key/业务数据/无]
拦截影响: [凭据泄露可横向攻击/仅业务数据泄露/无敏感数据]
```

**并行模式下，并行Agent必须返回传输安全信号和初步反向追踪结果，主Agent补充验证**。

**步骤4：工具辅助验证**

根据步骤1-3的结果，选择合适的工具进行验证：

| 风险类型 | 首选工具 | 验证方法 |
|---------|---------|---------|
| SSL验证关闭 | Burp Suite | 配置为代理，观察应用是否接受代理证书 |
| 明文传输 | Wireshark | 捕获网络流量，检查明文敏感数据 |
| CORS滥用 | curl/ZAP | 构造跨域请求，检查CORS响应头 |
| 凭据传输 | Burp Suite | 捕获认证请求，检查凭据传输方式 |
| 本地服务暴露 | nmap/curl | 扫描端口，测试认证 |

**重要**：工具验证仅在明确授权的测试环境中进行。生产环境仅基于源码证据分析。

**步骤5：传输安全证据闭合**

对每条传输链路风险，验证以下证据链：

```
传输源(Source): 敏感数据产生点（代码位置）
传输协议(Protocol): 使用的协议及版本
传输验证(Verification): SSL/TLS证书验证状态
传输防护(Protection): 传输层加密、签名、完整性校验
拦截可行性(Interceptability): 攻击者拦截传输数据的条件
利用前置条件(Preconditions): 攻击者必须控制或具备什么
传输影响(Transport Impact): 拦截后可获取的敏感信息或可执行的操作
```

**步骤6：与调用链追踪结果交叉验证**

将传输链路探测结果与阶段6的调用链追踪结果交叉验证：
1. 调用链中的敏感数据是否经过不安全的传输链路
2. 传输链路的SSL验证状态是否影响调用链的安全性
3. 代理配置是否使调用链中的安全防护失效

### 阶段8：证据闭合验证

对每条finding，必须证明以下证据链闭合：

```
源(Source): 外部输入来源
传播(Propagation): 经过的方法、字段、DTO、配置、数据库或消息路径
Sink: 危险操作
防护(Sanitizer): 强防护、弱防护、未知防护或无防护
守卫(Guard): 认证、角色、所有权、租户、业务约束
传输安全(Transport): 传输链路SSL/TLS验证状态、协议安全性、拦截可行性（来自阶段7）
利用前置条件(Preconditions): 攻击者必须控制或具备什么
影响(Impact): 实际影响
```

**调用链深度不是目标，证据闭合才是。** 一跳可达的漏洞如果证据闭合即可确认；长调用链如果关键证据缺失则不能确认。

**传输安全维度说明**：如果finding涉及网络通信（API调用、认证请求、数据传输等），必须评估传输链路安全性。传输安全为"不安全"时（如SSL验证关闭、明文传输），即使代码逻辑有防护，攻击者仍可在传输层绕过。传输安全评估结果来自阶段7的探测。

**每个维度必须标注证据状态**：
- ✅闭合：有明确源码证据
- ❌缺失：关键证据无法获取
- ⚠️推断：基于间接证据推断，非直接源码确认
- ➖不涉及：该finding不涉及网络传输等该维度的适用条件；必须说明为什么不涉及

**确认条件**：所有适用维度均为✅闭合，且不适用维度明确标为➖不涉及；任何适用维度为❌缺失或⚠️推断，均不得标记为“确认”。

### 阶段9：去重与报告

**去重**：
1. 同一漏洞点被多种关键词命中时合并为1条finding
2. 同一数据流上的多个危险点合并为1条finding，标注完整调用链
3. 检查遗漏：某类型Grep命中>10但finding=0，必须重新检查

**报告**：先按 `report-schema.md` 汇总并校验结构化finding，再生成Excel，包含"漏洞扫描报告"、"传输链路探测分析"、"扫描统计"和"跨代码仓调用链分析"四个Sheet。默认不输出P2/P3逐条候选，仅在统计中显示数量和处理摘要，并告知用户可选择导出候选附表。

**传输链路探测分析Sheet字段**：链路ID、传输源(代码位置)、目标端点、协议、传输数据类型、SSL/TLS验证状态、代理信任状态、拦截工具、拦截可行性、利用前置条件、传输影响、修复建议

**跨代码仓调用链分析Sheet字段**：链路ID、链路名称、调用方、被调用方、协议/通信方式、传输数据、安全风险点、跨代码仓漏洞链、攻击入口、攻击影响、严重等级、修复建议

## 结论状态

仅使用以下3种状态，**不再使用"可能"和"潜在"**：

- **确认(Confirmed)**：8维度证据全部闭合，且无强防护。报告中标记为"确认"的finding表示"真有问题"
- **误报(False Positive)**：明确证据证明输入不可控、数据流断裂或有强防护
- **缺失关键源码(Missing Source)**：关键证据无法获取（如防护实现未读取、调用链追踪不完整、传输内容未确认）。**不得直接判确认也不得直接判误报**，必须标注具体缺失环节

**关于"可能"和"潜在"**：如果证据不足以判"确认"，也不足以判"误报"，则归为"缺失关键源码"。这是对用户的诚实告知——"我无法确认这是否有问题，因为缺少XX证据"，而非给出模糊的"可能有问题"。

**缺失关键源码必须包含3部分信息**：

```
1. 缺失原因：为什么缺失关键源码？缺了什么？
   - 示例："防护函数fileNameCheck的实现代码不在当前代码仓中，无法确认是否为强防护"
   - 示例："调用链追踪到达Feign Client边界，被调用方Service的实现代码不可获取"
   - 示例："日志级别配置未找到，无法确认DEBUG日志是否在生产环境输出"

2. 缺失证据的影响：缺失的证据如何影响判断？
   - 示例："如果fileNameCheck是强防护（白名单校验），则该路径穿越可判误报；如果是弱防护（仅扩展名检查），则维持P1"
   - 示例："如果被调用方Service对输入做了参数化查询，则SQL注入不成立；如果直接拼接则确认风险"

3. 用户接力分析指引：用户接下来该怎么做？可能的判断方向和理由
   - 示例："建议读取FileCheckUtils.fileNameCheck的实现：1)若包含getCanonicalPath+startsWith基目录检查→强防护→可判误报；2)若仅检查扩展名→弱防护→维持P1→建议使用UUID重命名方案修复"
   - 示例："建议检查OrderService.processOrder方法：1)若使用#{}参数化查询→SQL注入不成立；2)若使用${}拼接→确认风险→修复为参数化查询"
   - 示例："建议确认logback-spring.xml中相关Logger的级别配置：1)若生产级别≥INFO→DEBUG日志不输出→降级为低风险；2)若DEBUG级别生效→确认敏感信息泄露风险"
```

这3部分信息让用户能够：理解为什么无法判定 → 评估缺失证据的两种可能走向 → 知道下一步该查什么代码、查到后如何判断。

## 误报证明要求

判为误报必须至少证明以下之一：
- 输入无法被外部攻击者或相关上游控制
- 源到Sink的数据流断裂
- 认可的强防护适用于该确切路径（**必须读取防护实现代码，不可凭函数名推断**）
- 命中在测试/演示/未部署代码中且无生产引用

**以下不能单独作为误报证明**：
- Payload执行失败
- 接口是服务间调用
- 辅助函数名看起来像验证但未读取实现
- 全局安全组件存在但未映射到该路径
- 命中数量大

## 防护措施分类

每种漏洞审查必须将防护分类为强防护、弱防护、未知防护或无防护。

### 分类定义

```
强防护: 映射到确切路径时，可证明误报或大幅降级
弱防护: 降低风险但不能单独证明安全
未知防护: 名称类似验证的包装器，实现必须读取——未读取则不可判误报，finding维持当前排序
无防护: 未发现相关防护
```

**关键规则**：未知防护不得判误报。必须读取实现代码后才能分类。如果无法读取实现，finding维持当前排序并标注"防护实现未确认"。

### 18种漏洞类型防护速查表

完整的18种漏洞类型防护分类速查表（含强防护/弱防护/未知防护/无防护的具体条目和说明）见 `shared-references/sanitizers.md`。

**加载时机**：主Agent在阶段5审查P0/P1候选时加载sanitizers.md，按漏洞类型查阅对应防护分类。并行Agent不需要加载此文件。

**速查表覆盖范围**：SQL注入、命令注入、路径穿越、SSRF、XSS、XXE、硬编码、敏感信息写入日志、权限控制、不安全反射、反序列化、CSV注入、表达式注入、弱随机数、网络安全配置、资源消耗、输入校验、供应链/签名校验缺失——共18种，确保任何项目扫描时都有判定依据。

## Grep截断处理规则

当Grep结果截断（N>100）时，按优先级依次执行，确保P0/P1候选不被遗漏：

1. **分目录搜索**：将path参数缩小到各子项目目录
2. **分文件类型搜索**：使用include参数（`*.java`/`*.xml`/`*.vue`/`*.properties`）
3. **更精确的pattern**：如SQL注入将`${`改为`ORDER BY ${`等
4. **不得跳过**：截断时必须通过上述方法确保高优先级候选不被遗漏

## 并行Agent输出格式（并行模式强制）

以 `agent-output-schema.md` 为机器可校验的字段契约；以下格式仅为人读摘要。主Agent先校验必填字段、追踪深度和未确认防护，再接收候选。

每个并行Agent必须返回以下结构化数据。主Agent基于这些数据执行阶段5-8。

### 1. 候选清单

## 大代码仓并行扫描策略

**核心原则：并行Agent只做发现和追踪，主Agent做判定和验证。子Skill只由主Agent加载，不蒸馏不压缩。**

### 两阶段扫描流程（强制）

**阶段A：并行发现与追踪**

- **3个以上子项目** → 必须使用Agent工具分派并行Agent扫描
- **单个子项目Java文件>500** → 按模块拆分并行
- 每个Agent负责1-2个子项目，执行阶段3(候选发现)+4(排序)+6(调用链追踪)
- 每个Agent返回结构化候选清单（见"并行Agent输出格式"），**不返回最终finding**
- 每个Agent还必须返回跨仓调用点清单和传输安全信号

**阶段B：主Agent深度审查与验证——不可跳过**

并行Agent返回结果后，主Agent必须执行以下步骤：

**步骤1：加载子Skill深度审查**
- 按漏洞类型逐个加载`skill/`下对应的子Skill
- 对每条P0/P1候选：
  - 读取候选代码上下文（并行Agent已提供位置）
  - 读取候选中提到的防护实现代码（从"防护未确认清单"中获取）
  - 按子Skill判定标准给出最终结论：确认/误报/缺失关键源码
  - 按子Skill证明模式编写结构化证明

**步骤2：防护实现确认**
- 读取"防护未确认清单"中的每个防护实现代码
- 按防护分类速查表判定强/弱/未知
- 更新对应候选的排序和证据闭合评估

**步骤3：跨代码仓关联分析**
- 收集各Agent的跨仓调用点清单，构建全局跨仓调用关系图
- 串联不同代码仓的候选，识别组合漏洞链（详见阶段6.5）
- 凭据横向扩展分析：识别共享凭据，评估一处泄露的横向影响
- 供应链攻击链分析：关注"代码仓A提供下载→代码仓B执行"模式

**步骤4：传输链路探测**
- 对并行Agent返回的传输安全信号，执行敏感数据反向追踪（见阶段7步骤3）
- 验证并行Agent的初步追踪结果
- 对未完成反向追踪的信号，主Agent补充追踪

**步骤5：证据闭合验证**
- 对每条finding，按8维度逐一验证闭合
- 每个维度标注个维度标注✅闭合/❌缺失/⚠️推断
- 只有8维度全部✅闭合的finding才能标记为"确认"

**为什么阶段B不可跳过**：
- 并行Agent无法加载子Skill，缺少完整判定标准和误报模式
- 并行Agent无法看到其他代码仓的上下文，防护分类可能不准
- 判定需要读取防护实现代码（如fileNameCheck、OgnlUtil），并行Agent可能跳过
- 传输链路敏感数据反向追踪需要全局视图，并行Agent只能看到局部
- "上传接口无签名校验"（代码仓A）+ "下载后直接执行"（代码仓B）= 供应链RCE，但任一Agent单独看都只是"正常业务逻辑"

## 扫描输出字段

编号、漏洞类别、漏洞标题、严重等级、置信度、漏洞位置(文件:行号)、核心问题、外部输入可控性、中间防护及绕过、证据闭合评估、利用前置条件、修复建议、验证结论、验证详情、完整调用链

**证据闭合评估字段格式**：
```
Source: ✅闭合 / ❌缺失 / ⚠️推断 — [具体证据]
Propagation: ✅闭合 / ❌缺失 / ⚠️推断 — [具体证据]
Sink: ✅闭合 / ❌缺失 / ⚠️推断 — [具体证据]
Sanitizer: ✅闭合 / ❌缺失 / ⚠️推断 — [强防护/弱防护/未知防护/无防护, 具体证据]
Guard: ✅闭合 / ❌缺失 / ⚠️推断 — [具体证据]
Transport: ✅闭合 / ❌缺失 / ⚠️推断 / 不涉及 — [具体证据]

Impact: ✅闭合 / ❌缺失 / ⚠️推断 — [具体证据]
```

修复建议应简明：给出方向和应避免的不安全做法，不设计完整业务实现。

**验证详情字段格式**：

缺失关键源码的验证详情必须包含3部分：
```

缺失证据的影响: [缺失的证据如何影响判断？两种可能走向及各自结论]
用户接力分析指引: [用户接下来该查什么代码？查到后如何判断？]
```

## 核心纪律（18条）

1. **不脑补未看到的事实**：代码中没有看见用户输入或防护，如实说明
2. **必须追踪完整数据流**：从用户输入到漏洞操作的完整路径，最少追踪2层调用链
3. **必须验证防护措施有效性**：区分强防护和弱防护，未知防护必须读取实现，不可凭函数名推断
4. **必须给出置信度**：不确定时明确标注
5. **必须完成证据闭合验证**：对每条finding证明Source→Propagation→Sink→Sanitizer→Guard→Transport→Preconditions→Impact证据链完整，每个适用维度标注✅/❌/⚠️；仅不适用维度可标➖，所有适用维度✅闭合才能标记"确认"
6. **不要说谎或捏造事实**：无法确认时标注"缺失关键源码"
7. **主Agent必须加载子Skill**：主Agent在阶段5审查P0/P1候选时，必须加载对应子Skill获取完整判定标准。并行Agent不需要加载子Skill
8. **必须处理Grep截断**：Grep结果截断时必须分目录/分文件类型二次搜索，确保P0/P1候选不被遗漏
9. **必须记录完整调用链**：每条finding必须包含从输入源到危险点的完整调用链
10. **缺失关键源码不得判误报**：追踪不完整时标注"缺失关键源码"，不得直接判定为"确认误报"
11. **大代码仓必须并行扫描**：3个以上子项目时必须使用Agent工具分派并行Agent
12. **必须执行传输链路评估**：调用链涉及网络通信时，必须评估传输链路安全性（阶段7）；仅在授权测试环境使用安全工具验证拦截可行性，source-only模式只记录源码分析推断
13. **工具探测是专业实践**：网络安全工程师使用Burp Suite、mitmproxy、Wireshark等工具验证发现是合理且必要的，不限于敏感信息泄露，覆盖认证绕过、会话劫持、请求篡改、CORS滥用、SSL降级等攻击面
14. **不脑补工具验证结果**：未实际使用工具验证时标注"源码分析推断"，已验证时标注"工具验证确认"
15. **发现verify=False/http://必须追踪传输内容**：不能只记录"SSL验证关闭"或"HTTP明文"而不追踪该HTTP客户端传输什么敏感数据。必须执行敏感数据反向追踪：信号 → HTTP客户端实例 → 所有请求 → 请求中的认证/凭据/敏感数据 → 拦截影响评估
16. **必须执行跨代码仓调用链追踪**：扫描目标包含2个以上子项目/代码仓时，必须执行跨代码仓调用链追踪（阶段6.5）。一个代码仓的输入验证漏洞可能通过跨代码仓调用链在另一个代码仓中被放大为RCE；传输安全漏洞在跨代码仓场景下影响倍增；共享凭据的多个代码仓必须评估凭据横向扩展风险
17. **并行Agent只做发现，主Agent做判定**：并行Agent返回结构化候选清单（含调用链和防护名称），主Agent加载子Skill后逐一审查判定。不得将判定职责委托给并行Agent
18. **结论只有三种**：确认(真有问题)、误报(确认没问题)、缺失关键源码(证据不足)。不使用"可能"和"潜在"——如果证据不足以判确认，归为"缺失关键源码"
