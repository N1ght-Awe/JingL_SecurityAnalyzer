## 1. 漏洞概述

配置项本身不是结论。必须证明它在可暴露部署路径生效，并评估受影响客户端实际传输的凭据或敏感数据。

## 2. 危险点清单

- `verify=False`、`CERT_NONE`、`check_hostname=False`、信任所有证书。
- `http://`、`ws://` 非回环地址；宽松 CORS+凭据；盲目信任代理头；`0.0.0.0` 管理端点。
- 缺少 `Secure`/`HttpOnly`/`SameSite`，关闭 CSRF/安全头。

## 3. 分析步骤

1. 读取 profile、环境覆盖、部署清单和网关边界，确定配置生效范围。
2. 从不安全 HTTP 客户端实例反向追踪所有请求、认证头、Cookie、请求体与响应敏感字段。
3. 对 CORS 同时核验 origin、methods、headers、credentials 和实际路由认证。
4. 仅在授权测试环境使用代理工具验证；未验证时标为源码分析推断。

## 4. 证据闭合验证

Source 为配置/客户端实例；Propagation 为请求构建；Sink 为网络通信或暴露端点；Sanitizer 为 TLS/精确 origin/安全 Cookie；Guard 为部署边界；Transport 必须记录；Impact 为 MITM、跨站读取、管理暴露。

## 5. 防护措施清单

**强防护：** 生产强制证书/主机验证、HTTPS/WSS、精确 CORS allowlist、最小代理信任、受保护管理端点。

**弱防护：** “仅开发环境”但无隔离证据、仅 CDN/网关补偿、`*` origin 与凭据组合。

## 6. 常见误报模式

仅测试 profile 且部署隔离可证明；回环无代理且不传敏感数据；配置被生产覆盖且能定位覆盖证据。

## 7. 判定标准

确认：不安全配置在可达路径生效并暴露可利用影响，TLS/明文场景还须证明传输内容。误报：精确部署与客户端证据表明无影响。缺少部署或调用点时标记缺失关键源码。

## 8. 置信度评估

高：生效配置+敏感传输/暴露端点；中：源码配置未确认环境；低：关键词命中。

## 9. 核心纪律

发现 `verify=False` 或明文 URL 后必须完成敏感数据反向追踪，不得只报配置名称。
