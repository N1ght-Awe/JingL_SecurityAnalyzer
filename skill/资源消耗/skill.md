## 1. 漏洞概述

资源密集 API 不是结论。需证明攻击者可影响资源规模或循环次数，且路径缺少适当大小、时间、深度、速率或并发限制。

## 2. 危险点清单

- `readAllBytes`、`toByteArray`、全量列表/分页缺失、无限循环/递归。
- 文件上传、Zip/Gzip 解压、图像/文档解析、正则回溯、批量导出、远程拉取。

## 3. 分析步骤

1. 追踪大小、条目数、递归深度、分页参数、压缩比和超时的来源。
2. 阅读所有限制的实际执行位置：网关、框架、控制器、流式读取、任务队列。
3. 检查单请求限制之外的并发、速率、磁盘配额和取消机制。
4. 对压缩包分别核验压缩前大小、解压后大小、条目数、路径和嵌套层数。

## 4. 证据闭合验证

Source 为不可信规模参数/内容；Propagation 为资源分配；Sink 为无界操作；Sanitizer 为可验证限制；Guard 为限流/配额；Impact 为服务不可用。

## 5. 防护措施清单

**强防护：** 在消耗前和消耗中执行最大大小、条目、深度、时间与并发限制，失败可控。

**弱防护：** 仅 Content-Length、仅前端限制、仅单次超时、限制值未在实际读取循环生效。

## 6. 常见误报模式

输入大小由不可绕过网关限制且业务层再次限制；后台固定数据批处理；资源操作有严格上界。

## 7. 判定标准

确认：外部可控规模到达无界/可放大资源操作，且无有效限制。误报：完整路径强制上界。配置或入口缺失时标记缺失关键源码。

## 8. 置信度评估

高：可控上传/压缩包直达无界操作；中：限制实现未知；低：仅资源 API。

## 9. 核心纪律

不要用“框架默认限制”作证明；定位配置、运行 profile 与实际适用路由。
