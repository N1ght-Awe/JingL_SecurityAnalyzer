本文件定义最终finding的证据要求、结论状态和报告字段。

## 证据闭合

正式finding必须覆盖：

```
源(Source): 外部输入来源
传播(Propagation): 经过的方法、字段、DTO、配置、数据库或消息路径
Sink: 危险操作
防护(Sanitizer): 强防护、弱防护、未知防护或无防护
守卫(Guard): 认证、角色、所有权、租户、业务约束
传输安全(Transport): 传输链路SSL/TLS验证状态、协议安全性、拦截可行性（仅当finding涉及网络通信时）
利用前置条件(Preconditions): 攻击者必须控制或具备什么
影响(Impact): 实际影响
```

调用链深度不是目标。一跳可达的漏洞如果证据闭合即可确认；长调用链如果关键证据缺失则不能确认。

**传输安全(Transport)维度说明**：如果finding涉及网络通信（API调用、认证请求、数据传输等），必须评估传输链路安全性。传输安全评估来自阶段7（安全工具辅助传输链路探测）的结果。传输安全为"不安全"时（如SSL验证关闭、明文传输），即使代码逻辑有防护，攻击者仍可在传输层绕过，应升级严重等级或增加前置条件说明。

## 结论状态

仅使用以下3种状态，不使用"可能"和"潜在"：

- **确认(Confirmed)**：8维度证据全部闭合（Source→Propagation→Sink→Sanitizer→Guard→Transport→Preconditions→Impact），且无强防护。报告中标记为"确认"的finding表示"真有问题"
- **误报(False Positive)**：明确证据证明输入不可控、数据流断裂或强防护适用于该确切路径（强防护必须读取实现代码确认，不可凭函数名推断）
- **缺失关键源码(Missing Source)**：关键证据无法获取（如防护实现未读取、调用链追踪不完整、传输内容未确认），不得直接判确认也不得直接判误报，必须标注具体缺失环节

**缺失关键源码必须包含3部分信息**：

1. **缺失原因**：为什么缺失关键源码？缺了什么？
2. **缺失证据的影响**：缺失的证据如何影响判断？（两种可能走向及各自结论）
3. **用户接力分析指引**：用户接下来该查什么代码？查到后如何判断？

示例：
```
缺失原因: 防护函数fileNameCheck的实现代码不在当前代码仓中，无法确认是否为强防护
缺失证据的影响: 如果fileNameCheck是强防护(白名单校验)→可判误报；如果是弱防护(仅扩展名检查)→维持P1
用户接力分析指引: 建议读取FileCheckUtils.fileNameCheck的实现：1)若包含getCanonicalPath+startsWith→强防护→判误报；2)若仅检查扩展名→弱防护→维持P1→建议UUID重命名修复
```

**关于"可能"和"潜在"**：如果证据不足以判"确认"，也不足以判"误报"，则归为"缺失关键源码"。这是对用户的诚实告知——"我无法确认这是否有问题，因为缺少XX证据"，而非给出模糊的"可能有问题"。

## 证据维度状态标注

每个finding的8个证据维度必须标注状态：
- ✅闭合：有明确源码证据
- ❌缺失：关键证据无法获取
- ⚠️推断：基于间接证据推断，非直接源码确认

**适用性规则**：Transport只在涉及网络通信时适用；不涉及网络通信时必须标注“➖不涉及”并说明通信节点不存在。其他维度通常适用；如业务模型明确不适用，必须写出理由。

**确认条件**：所有适用维度均为✅闭合，且不存在强防护。任何适用维度为❌缺失或⚠️推断，均不得标记为“确认”；❌缺失归为“缺失关键源码”。⚠️推断必须继续追踪，无法转为直接证据时归为“缺失关键源码”。

**优先级、严重等级、置信度与结论必须分离**：P0/P1/P2/P3仅决定审查优先级；严重等级基于实际影响、暴露范围和利用前置条件；置信度基于证据直接性和完整性；结论仅由证据闭合与防护适用性决定。

## 间接输入与跨仓缺失环节

MQ、数据库二次读取、缓存、文件、RPC参数和其他代码仓输出均是当前仓的信任边界外数据，不能因“服务间调用”而视为可信。

- 当前仓能证明该数据未经过适当边界校验即到达危险Sink时，保留P1并继续追踪。
- 未提供生产者仓或上游入口时，不得声称“攻击者可控”或端到端可利用；结论为“缺失关键源码”。
- 验证详情必须直白说明：`不知该数据是否攻击者可控；缺少[生产者仓/写入端/调用方]源码。若攻击者可影响该数据，则当前路径可到达[Sink]；若上游已强校验且不可绕过，则该finding可判误报或降级。`

只有当前仓自身存在直接外部入口，或所提供的上游仓已将攻击者可控性闭合时，才可将这类链路标为“确认”。

## 动态严重等级

索引中的CVSS仅作类型风险上限参考，不是最终分数。主Agent按以下事实动态定级并说明理由：

- **影响**：RCE、跨租户数据访问、凭据泄露、完整性破坏、拒绝服务的实际范围；
- **暴露面**：未认证、普通已认证、特权账户、内网或仅本地；
- **前置条件**：攻击者需要的权限、网络位置、受害者交互、可控数据来源；
- **可达性与防护**：调用链是否完整、强防护是否生效、传输层是否可绕过。

报告中的严重等级必须针对该finding动态给出；如同时给出CVSS，记录采用的向量或调整理由。

## 误报证明要求

判为误报必须至少证明以下之一：
- 输入无法被外部攻击者或相关上游控制
- 源到Sink的数据流断裂
- 认可的强防护适用于该确切路径（**必须读取防护实现代码确认，不可凭函数名推断**）
- 命中在测试/演示/未部署代码中且无生产引用

**以下不能单独作为误报证明**：
- Payload执行失败
- 接口是服务间调用
- 辅助函数名看起来像验证但未读取实现
- 全局安全组件存在但未映射到该路径
- 命中数量大

security-tool-probing.md 理论上的安全工具闭合内容 # 安全工具辅助传输链路探测

本文件定义安全工具辅助验证方法论。作为网络安全工程师，动用工具发现问题、搜索问题、用工具闭环调用链是合理且必要的专业实践。

## 设计原则

1. **工具是验证手段，不是目的**：工具探测用于补充源码分析的不足，验证传输链路上的安全风险
2. **不限于单一工具**：根据场景选择最合适的工具，Burp Suite、mitmproxy、Wireshark、OWASP ZAP等各有适用场景
3. **不限于敏感信息泄露**：传输链路探测覆盖认证绕过、会话劫持、请求篡改、CORS滥用、SSL降级等多种攻击面
4. **探测结果纳入证据闭合**：工具探测的发现作为证据链中"传输安全(Transport)"维度的输入

## 工具矩阵

| 工具 | 适用场景 | 探测能力 | 使用方式 |
|------|---------|---------|---------|
| **Burp Suite** | HTTP/HTTPS应用层拦截 | 请求/响应篡改、重放、参数注入、认证绕过验证 | 透明代理模式，配置浏览器或应用指向代理 |
| **OWASP ZAP** | HTTP/HTTPS自动化扫描 | 主动扫描、被动扫描、API模糊测试、认证测试 | 自动化扫描模式或手动代理模式 |
| **mitmproxy** | HTTP/HTTPS程序化拦截 | 脚本化请求/响应修改、批量流量分析、自动化回归 | 命令行或Python脚本驱动 |
| **Wireshark/tcpdump** | 网络层包捕获 | 明文协议分析、SSL/TLS握手检查、DNS查询、网络拓扑发现 | 网络接口捕获或pcap文件分析 |
| **Browser DevTools** | 客户端网络检查 | 请求头/响应头、Cookie属性、WebSocket帧、混合内容 | F12开发者工具Network/Security面板 |
| **Postman/curl** | API请求构造与重放 | 认证绕过测试、参数篡改、CSRF验证、速率限制测试 | 手动或集合自动化 |
| **nmap** | 服务发现与端口扫描 | 开放端口、服务版本、SSL/TLS配置、证书信息 | 命令行扫描 |
| **testssl.sh** | SSL/TLS配置审计 | 协议版本、密码套件、证书链、HSTS、已知漏洞 | 命令行对目标服务执行 |

## 探测维度与检查项

### 维度1：SSL/TLS证书验证

**源码信号**：
- `verify=False`、`ssl_verify=False`、`verify=False`（httpx/requests）
- `ignore_https_errors=True`（Playwright/Puppeteer）
- `InsecureRequestWarning` 被抑制
- `ssl._create_unverified_context()`
- `TrustManager` 自定义实现接受所有证书
- `check_hostname=False`

**工具验证**：
1. 使用Burp Suite/ZAP配置为代理，观察应用是否接受代理的自签名证书
2. 使用testssl.sh检查目标服务的TLS配置
3. 使用Wireshark捕获握手过程，检查证书链验证

**风险判定**：
- 应用接受任意证书 → 传输链路可被中间人拦截
- 仅开发环境禁用验证 → 检查是否有环境隔离机制
- 配置项可被运行时覆盖 → 追踪实际生效值

### 维度2：明文协议传输

**源码信号**：
- `http://`（非`https://`）的API端点配置
- WebSocket使用`ws://`而非`wss://`
- 本地HTTP代理/服务（127.0.0.1上的HTTP服务）
- 开发态代理（如DevLLMProxy、webpack-dev-server）

**工具验证**：
1. 使用Wireshark捕获本地网络接口，观察是否有明文敏感数据
2. 使用curl -v检查HTTP请求中的敏感头部和载荷
3. 使用mitmproxy脚本自动化检测明文传输的敏感字段

**风险判定**：
- 本地回环(127.0.0.1)上的HTTP → 低风险（需本地访问）
- 局域网HTTP → 中风险（ARP欺骗可拦截）
- 公网HTTP → 高风险（任何网络节点可拦截）

### 维度3：认证凭据传输安全

**源码信号**：
- Cookie缺少`Secure`/`HttpOnly`/`SameSite`属性
- Authorization头部通过HTTP传输
- 认证Token在URL参数中传递
- CSRF Token在跨域请求中暴露
- 硬编码的API Key/Secret在请求中发送

**工具验证**：
1. 使用Burp Suite代理捕获登录/认证请求，检查凭据传输方式
2. 使用DevTools检查Cookie属性和Set-Cookie头部
3. 使用ZAP的认证测试功能验证会话管理

**风险判定**：
- 凭据通过HTTPS+安全Cookie传输 → 低风险
- 凭据通过HTTP传输或Cookie缺少安全属性 → 高风险
- 凭据在URL中传递 → 极高风险（日志/Referer泄露）

### 维度4：CORS与跨域安全

**源码信号**：
- `allow_origins=["*"]` + `allow_credentials=True`
- `Access-Control-Allow-Origin: *` + `Access-Control-Allow-Credentials: true`
- 反射型CORS（`Access-Control-Allow-Origin`设置为请求的Origin值）

**工具验证**：
1. 使用Burp Suite构造跨域请求，观察CORS头部响应
2. 使用curl模拟跨域请求：`curl -H "Origin: https://evil.com" -v https://target/api`
3. 使用ZAP的CORS扫描插件

**风险判定**：
- `allow_origins=*` + `allow_credentials=True` → 高风险（CORS滥用）
- 反射型CORS → 高风险（任意域窃取数据）
- `allow_origins=*` 无凭据 → 中风险（数据暴露但无认证上下文）

### 维度5：代理与中间人攻击面

**源码信号**：
- 应用支持显式代理配置（HTTP_PROXY/HTTPS_PROXY环境变量）
- 应用使用系统代理设置
- 代理认证缺失或弱认证
- 代理转发时未验证上游证书

**工具验证**：
1. 配置Burp Suite为系统代理，观察应用是否通过代理发送请求
2. 使用mitmproxy检查代理认证要求
3. 使用Wireshark检查代理连接是否使用CONNECT隧道

**风险判定**：
- 应用信任系统代理且不验证上游证书 → 高风险
- 代理仅用于开发环境且有环境隔离 → 中风险
- 代理配置需要显式启用且验证上游证书 → 低风险

### 维度6：本地服务暴露

**源码信号**：
- 本地HTTP服务（127.0.0.1或0.0.0.0绑定）
- 本地WebSocket服务
- 本地gRPC/Thrift服务
- 开发工具端口（调试器、热重载、DevTools协议）

**工具验证**：
1. 使用nmap扫描本地端口：`nmap -sT 127.0.0.1 -p-`
2. 使用curl测试本地服务认证：`curl -v http://127.0.0.1:PORT/api`
3. 使用Burp Suite通过代理链访问本地服务

**风险判定**：
- 本地服务无认证 + 0.0.0.0绑定 → 高风险（局域网可访问）
- 本地服务无认证 + 127.0.0.1绑定 → 中风险（需本地代码执行）
- 本地服务有认证 → 低风险

## 探测流程

### 步骤1：传输链路清单构建

从阶段6（调用链深度追踪）的结果中，提取所有涉及网络通信的调用链，构建传输链路清单：

```
链路ID: TL-001
源端: 应用内部（认证模块）
协议: HTTPS
目标: IDaaS SSO服务
传输数据: mtoken、authToken、Cookie
SSL验证: verify=False（配置项）
代理支持: 是（系统代理）
风险标记: SSL验证关闭 + 凭据传输
```

对每条传输链路记录：
- 链路ID
- 源端和目标端
- 协议（HTTP/HTTPS/WS/WSS/TCP）
- 传输的敏感数据类型
- SSL/TLS验证状态
- 代理支持情况
- 初步风险标记

### 步骤2：源码信号扫描

在阶段3（轻量候选发现）中增加传输链路相关的Grep关键词扫描（见scan-index.md更新），识别：
- SSL验证关闭的代码点
- 明文协议使用
- 不安全的Cookie/Token处理
- CORS配置问题
- 代理信任配置

### 步骤3：工具辅助验证

根据步骤1和2的结果，选择合适的工具进行验证：

| 风险类型 | 首选工具 | 验证方法 |
|---------|---------|---------|
| SSL验证关闭 | Burp Suite | 配置为代理，观察应用是否接受代理证书 |
| 明文传输 | Wireshark | 捕获网络流量，检查是否有明文敏感数据 |
| CORS滥用 | curl/ZAP | 构造跨域请求，检查CORS响应头 |
| 凭据传输 | Burp Suite | 捕获认证请求，检查凭据传输方式 |
| 本地服务暴露 | nmap/curl | 扫描端口，测试认证 |

**重要**：工具验证仅在明确授权的测试环境中进行。生产环境仅基于源码证据分析。

### 步骤4：传输安全证据闭合

对每条传输链路风险，验证以下证据链：

```
传输源(Source): 敏感数据产生点（代码位置）
传输协议(Protocol): 使用的协议及版本
传输验证(Verification): SSL/TLS证书验证状态
传输防护(Protection): 传输层加密、签名、完整性校验
拦截可行性(Interceptability): 攻击者拦截传输数据的条件
利用前置条件(Preconditions): 攻击者必须控制或具备什么
传输影响(Transport Impact): 拦截后可获取的敏感信息或可执行的操作
```

### 步骤5：与调用链追踪结果交叉验证

将传输链路探测结果与阶段6的调用链追踪结果交叉验证：
1. 调用链中的敏感数据是否经过不安全的传输链路
2. 传输链路的SSL验证状态是否影响调用链的安全性
3. 代理配置是否使调用链中的安全防护失效

## 核心纪律

1. **工具探测是专业实践**：网络安全工程师使用工具验证发现是合理且必要的
2. **不限于Burp Suite**：根据场景选择最合适的工具
3. **不限于敏感信息泄露**：传输链路探测覆盖认证绕过、会话劫持、请求篡改等多种攻击面
4. **源码优先，工具验证**：先基于源码识别风险点，再用工具验证可利用性
5. **仅授权环境使用工具**：生产环境仅基于源码证据分析
6. **传输安全纳入证据闭合**：传输链路安全作为证据链的必要维度
7. **不脑补工具验证结果**：未实际使用工具验证时标注"源码分析推断"，已验证时标注"工具验证确认"
