# 防护措施分类

每种漏洞审查必须将防护分类为强防护、弱防护、未知防护或无防护。

## 分类定义

```
强防护: 映射到确切路径时，可证明误报或大幅降级
弱防护: 降低风险但不能单独证明安全
未知防护: 名称类似验证的包装器，实现必须读取——未读取则不可判误报，finding维持当前排序
无防护: 未发现相关防护
```

**关键规则**：未知防护不得判误报。必须读取实现代码后才能分类。如果无法读取实现，finding维持当前排序并标注"防护实现未确认"。

**分类生效门槛**：下表中的“强”是可验证的强防护模式，不是函数名、注解名或框架名的自动结论。只有同时满足“实现已读取、配置/参数已确认、映射到确切Source→Sink路径”时，才可按强防护处理；任一条件不满足时归为未知。框架默认值、环境配置和部署边界同样需要证据。

**同一数据路径规则**：防护必须处理到达当前Sink的同一可控字段、同一编码/序列化形态和同一调用分支。保护硬编码字段B、另一DTO字段、另一接口或另一分支，不能证明用户输入A安全；这种“局部使用框架防护、关键输入遗漏”的情形按无防护或未知防护处理，绝不因项目中存在框架组件而判误报。

## 18种漏洞类型防护速查表

> 速查表基于Java/Spring Boot/Jalor技术栈编写。项目可以不涉及某种漏洞类型，但速查表必须覆盖全部18种，确保任何项目扫描时都有判定依据。非Java项目（Python/Go/前端）参照同类防护语义映射。

### 1. SQL注入

| 防护 | 分类 | 说明 |
|------|------|------|
| `#{}`参数化查询 | 强 | PreparedStatement参数绑定 |
| 后端枚举映射(Enum.valueOf) | 强 | 用户输入只能映射为预定义固定值 |
| Map白名单(sortMapping.get) | 强 | 外部名称映射到固定SQL标识符 |
| Switch常量映射 | 强 | 编译期确定的安全值 |
| OgnlUtil.getOrderType() | 未知（满足条件后为强） | 读取实现并确认是白名单映射后才成立 |
| LambdaQueryWrapper方法引用 | 强 | 编译期确定列名 |
| 黑名单关键词过滤 | 弱 | 可绕过 |
| 正则过滤(未绑定业务白名单) | 弱 | 可能遗漏注入向量 |
| 长度限制 | 弱 | 不阻止短注入片段 |
| 字符替换 | 弱 | 双写绕过 |

### 2. 命令注入

| 防护 | 分类 | 说明 |
|------|------|------|
| 命令参数白名单/枚举 | 强 | 用户输入映射到固定命令 |
| 命令参数不来自用户输入 | 强 | 硬编码或配置注入 |
| ProcessBuilder仅传常量参数 | 强 | 编译期确定 |

| 仅长度限制 | 弱 | 不阻止短注入片段 |
| shell=True+用户输入 | 无 | 直接命令注入 |

## 传输链路安全防护（补充）

### 传输链路安全

强防护：
- SSL/TLS证书验证启用且硬编码不可关闭
- 所有敏感数据传输使用HTTPS/WSS
- HSTS头部启用
- 证书钉扎(Certificate Pinning)
- 双向TLS认证(mTLS)
- 代理不可绕过证书验证

弱防护：
- SSL验证可通过配置项关闭（如verify=False配置）
- 开发环境禁用SSL验证但无环境隔离
- 本地HTTP服务(127.0.0.1)传输敏感数据
- 代理信任系统证书存储
- Cookie缺少Secure属性但使用HTTPS

### 敏感数据传输（传输链路敏感数据交叉分析）

强防护：
- 敏感数据不在传输链路中出现（本地计算、零知识证明等）
- 敏感数据传输前加密，密钥不通过同一链路传输
- 敏感数据使用短期一次性Token替代长期密钥
- 远程配置API不返回密钥明文，仅返回引用或加密值
- 客户端凭证（如API Key）仅存储在安全存储（Keyring/Vault）中，不在网络请求中传输

弱防护：
- 敏感数据通过安全链路传输但存储在可访问的位置（环境变量、.env文件）
- 远程配置API返回密钥但通过HTTPS传输（SSL验证仍可被关闭覆盖）
- 使用Keyring存储但同一进程内存中仍有明文
- API Key通过请求头传输但客户端SSL验证可配置关闭
